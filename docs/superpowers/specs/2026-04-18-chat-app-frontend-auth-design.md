# Chat App Frontend And Auth Design

## Goal

为多智能体旅行规划助手补齐“前端页面与账号体系”的第一版能力。前端采用普通用户聊天应用风格：用户登录后直接进入全屏聊天体验，通过自然语言规划旅行；历史会话、方案详情和账号入口围绕聊天流展开，而不是做成后台工作台。

## Product Direction

第一版面向普通旅行用户，核心体验是“像和私人旅行助理聊天一样规划行程”。页面优先降低使用门槛，避免管理后台式信息密度。用户进入应用后应能立刻看到输入框、示例问题和历史入口。

界面主线：

1. 用户注册或登录。
2. 进入聊天首页。
3. 输入旅行需求。
4. 前端创建异步任务并展示生成进度。
5. 助手返回摘要回复。
6. 用户打开方案详情抽屉查看结构化行程。
7. 用户继续通过聊天或快捷操作调整方案。
8. 登录用户可以恢复自己的历史会话和方案。

## Scope

本次设计覆盖：

- 普通用户聊天应用风格 Web 前端。
- 登录、注册、登出、当前用户信息。
- 登录态持久化和未登录路由保护。
- 用户级会话隔离：会话和方案历史归属于当前用户。
- 聊天首页、历史侧栏、方案详情抽屉、账号入口。
- 与现有 FastAPI 对话、任务、方案接口的衔接。

## Out Of Scope

第一版不做第三方登录、手机号验证码、找回密码、会员权限、管理员后台、支付下单、真实订单交易闭环、复杂前端数据看板、移动原生 App。移动端通过响应式 Web 支持。

## Frontend Architecture

推荐新增 `frontend/` 目录，使用 React + Vite + TypeScript 构建单页应用。前端通过 REST API 调用当前 FastAPI 后端。

主要模块：

- `auth`: 登录、注册、token 存储、当前用户状态。
- `chat`: 消息流、输入框、异步任务轮询、错误重试。
- `sessions`: 历史会话列表、新建会话、切换会话。
- `plans`: 方案摘要、方案详情抽屉、局部重规划入口。
- `api`: 对后端接口的 typed fetch 封装。
- `ui`: 基础组件、布局、按钮、表单、抽屉、toast。

首版状态管理使用 React 内置 state 和 context，避免引入复杂全局状态库。后续如果页面复杂度上升，再考虑 TanStack Query 或 Zustand。

## Page Design

### Auth Pages

登录页包含用户名或邮箱、密码、登录按钮、跳转注册入口。注册页包含用户名、邮箱、密码、确认密码和注册按钮。表单字段必须有显式 label，错误显示在字段附近。

登录成功后跳转聊天首页。登录失败显示清晰原因，例如“账号或密码不正确”。Token 过期时清除本地登录态并跳转登录页。

### Chat Home

聊天首页是主页面，采用聊天应用布局：

- 顶部栏：应用名称、当前会话标题、历史按钮、账号菜单。
- 消息区：用户消息、助手消息、追问卡片、方案摘要卡片、任务进度。
- 底部输入区：多行输入框、发送按钮、新建规划入口。
- 历史侧栏：桌面端从左侧展开，移动端作为覆盖层展示。
- 方案详情抽屉：桌面端从右侧展开，移动端从底部或全屏覆盖。

空状态展示可点击示例问题，例如：

- “帮我规划杭州三日游，预算 3000。”
- “我想带父母去成都玩 4 天，节奏轻松一点。”
- “周末从上海出发，推荐一个高铁能到的目的地。”

### Plan Drawer

方案详情抽屉展示结构化行程，而不是把长方案全部塞进聊天气泡。内容包括：

- 方案概览：目的地、日期、天数、人数、预算。
- 交通推荐：候选方案、耗时、价格、推荐理由。
- 酒店推荐：区域、价格、评分、推荐理由。
- 每日行程：按 Day 1、Day 2 分组。
- 预算估算：交通、住宿、餐饮、门票、总计。
- 注意事项：天气、开放时间、交通提醒、风险提示。
- 快捷调整：改预算、换酒店、放慢节奏、增加景点、重新规划。

快捷调整调用 `/plan/{session_id}/revise`，并把调整结果回写当前聊天流和方案抽屉。

## Backend Auth Design

后端新增基础账号体系：

- `users` 表：用户 ID、用户名、邮箱、密码哈希、创建时间、更新时间。
- 注册接口：创建用户，用户名和邮箱唯一。
- 登录接口：校验密码并签发访问 token。
- 当前用户接口：根据 token 返回用户信息。
- 登出：前端删除 token；后端第一版不维护 token blacklist。

密码使用安全哈希算法保存，不保存明文。Token 使用 JWT 或等价签名 token，包含用户 ID、过期时间和签发时间。

会话、任务、方案历史需要和用户关联。第一版可以在现有 session/task/plan 查询入口增加用户校验，确保用户只能访问自己的会话与方案。

## API Contract

新增账号接口：

- `POST /auth/register`
- `POST /auth/login`
- `GET /auth/me`

前端继续使用已有旅行规划接口：

- `POST /chat/async`
- `GET /task/{task_id}`
- `GET /task/{task_id}/steps`
- `GET /session/{session_id}`
- `GET /plan/{session_id}`
- `GET /plan/{session_id}/history`
- `POST /plan/{session_id}/revise`

所有需要用户身份的接口通过 `Authorization: Bearer <token>` 传递登录态。

## Interaction States

聊天页必须覆盖这些状态：

- 未开始：展示示例问题。
- 输入中：输入框支持换行，发送按钮可用。
- 发送中：按钮进入 loading，防止重复提交。
- 任务运行中：展示任务进度，例如“正在理解需求”“正在生成交通方案”“正在整理行程”。
- 需要补充信息：展示追问卡片，用户点击后可快速回复。
- 成功：展示助手摘要和“查看完整方案”。
- 失败：展示错误原因和重试按钮。
- 登录过期：跳转登录页。
- 历史为空：提示开始第一次旅行规划。

## Visual Design

整体视觉走轻量、温和、可信的普通聊天应用风格：

- 浅色背景为主，深色文字保证阅读。
- 聊天气泡和方案卡片有清晰层级。
- 主要按钮使用一种鲜明但不过度刺激的主色。
- 圆角控制在 8px 或以下。
- 不使用 emoji 作为结构图标。
- 避免大面积紫蓝渐变、深蓝/ slate 单色主题、米色/棕色主导主题。
- 字体最小 16px，行高至少 1.5。
- 所有可交互元素有明显 hover、focus、disabled 和 loading 状态。

## Responsive Design

桌面端：

- 中间为聊天主栏。
- 历史作为左侧侧栏。
- 方案作为右侧抽屉。

移动端：

- 默认全屏聊天。
- 历史从侧边覆盖。
- 方案详情以底部抽屉或全屏详情展示。
- 输入框固定在底部，并避免被软键盘遮挡。
- 不允许横向滚动。

## Accessibility

必须满足：

- 表单使用真实 label。
- 按钮和图标按钮有可读名称。
- 键盘可操作登录、发送、打开历史、打开方案。
- focus 样式可见。
- 错误消息靠近对应字段。
- 加载和错误状态使用 `aria-live` 或等价方式提示。
- 不只依赖颜色表达状态。
- 正文和按钮文字对比度满足 WCAG AA。

## Testing

前端测试覆盖：

- 未登录访问聊天页会跳转登录页。
- 注册、登录成功后进入聊天页。
- 输入消息后调用 `/chat/async` 并显示任务进度。
- 任务成功后展示助手摘要和方案入口。
- 打开方案抽屉后展示交通、酒店、行程、预算模块。
- Token 过期时清理登录态并跳转登录页。

后端测试覆盖：

- 注册成功和重复邮箱失败。
- 登录成功和密码错误失败。
- `/auth/me` 在有 token 时返回用户。
- 未登录访问用户会话失败。
- 用户不能读取其他用户的 session、task 或 plan。

## Rollout

先实现后端账号接口和用户隔离，再实现前端登录注册，最后实现聊天首页和方案抽屉。每一步保持现有旅行规划 API 可用，避免破坏已有测试。
