# 多智能体旅行规划助手：需求开发总结

## 1. 项目定位

本项目是一个基于大模型与多智能体协同的旅行规划系统，面向用户自然语言旅行需求，支持通过多轮对话完成需求理解、任务拆解、信息检索、方案生成与持续调整，最终输出可执行的旅行方案。

项目当前确定的核心架构为：

- 1 个主 Agent：Planner / 主助手 Agent
- 3 个子 Agent：
  - Transport Agent（航班 / 交通 Agent）
  - Hotel Agent（酒店 Agent）
  - Itinerary Agent（行程 / 景点 Agent）

其中天气、地图、搜索等能力通过 tool 或 MCP 服务接入，不单独作为 Agent。

---

## 2. 项目目标

构建一个可扩展、可迭代、可工程化落地的多智能体旅行规划助手，具备以下核心能力：

- 支持自然语言输入旅行需求
- 支持多轮对话与状态记忆
- 能自动拆解交通、酒店、行程等任务
- 能调用工具或 MCP 服务获取外部信息
- 能生成结构化、可调整的旅行方案
- 能作为 FastAPI 后端服务独立运行
- 第一版先以 mock tools 实现最小可运行版本

---

## 3. Agent 架构设计

## 3.1 主 Agent

### Planner / 主助手 Agent
职责包括：

- 理解用户旅行需求
- 提取结构化字段
- 识别当前问题所属模块
- 拆解任务并路由到子 Agent
- 汇总子 Agent 返回结果
- 负责追问、澄清与重规划
- 统一维护会话状态与 trip_state

---

## 3.2 子 Agent

### 1. Transport Agent
负责：

- 识别交通相关需求
- 查询航班 / 高铁 / 火车候选方案
- 根据价格、时长、时间段等条件做筛选
- 给出交通推荐及理由
- 响应用户对交通方案的修改需求

### 2. Hotel Agent
负责：

- 识别住宿需求
- 查询酒店候选项
- 根据预算、地段、评分、交通便利度等条件筛选
- 输出推荐酒店及理由
- 响应用户对酒店方案的替换和调整

### 3. Itinerary Agent
负责：

- 检索景点、攻略、玩法信息
- 理解用户兴趣与旅行节奏偏好
- 结合旅行天数生成每日 itinerary
- 使用天气、地图、RAG 等工具优化行程顺序
- 响应用户对行程安排的增删改需求

---

## 4. 工具与 MCP 服务设计

以下能力不作为独立 Agent，而是作为子 Agent 可调用的外部能力：

- 天气查询工具
- 地图 / 路线工具
- 搜索工具
- 航班 / 高铁 / 火车查询工具
- 酒店查询工具
- RAG 检索工具
- Agent-Reach 外部信息检索工具
- 其他后续接入的 MCP 服务

设计原则：

- Agent 负责决策
- Tool / MCP 负责提供外部数据或执行外部能力
- 各子 Agent 内部按需调用工具，而不是继续拆出更多业务 Agent

## 4.1 Agent-Reach 接入定位

本项目可以借助 Agent-Reach 项目的搜索、网页读取、社交平台读取、视频字幕提取、GitHub / RSS / 公众号读取等能力，作为旅行规划助手的外部信息检索补充层。

Agent-Reach 在本项目中的定位不是新的业务 Agent，而是 tool / MCP 能力来源或适配层：

- 为 Itinerary Agent 提供景点攻略、玩法文章、本地经验、公开视频字幕、RSS / 公众号内容等检索来源
- 为 Hotel Agent 补充酒店周边、商圈、用户口碑、交通便利度等非结构化信息
- 为 Transport Agent 补充交通政策、出行提示、机场 / 车站信息、临时变更公告等公开信息
- 为 Planner Agent 提供综合检索能力，用于发现用户需求中的缺失信息、冲突信息和需要追问的点

建议在工程实现上新增统一的 `ExternalSearchTool` 或 `AgentReachSearchTool` 抽象，屏蔽 Agent-Reach 底层命令差异。子 Agent 只依赖统一工具接口，不直接耦合具体命令。

可优先接入的 Agent-Reach 能力包括：

- 网页阅读：使用 Jina Reader 读取旅游攻略、景点说明、官方页面等网页内容
- 全网搜索：使用 Exa / MCP 搜索目的地、景点、攻略、交通信息
- 视频字幕：使用 yt-dlp 获取 YouTube / B站旅游视频字幕，用于总结玩法和注意事项
- RSS / 公众号：读取旅游攻略、城市活动、目的地资讯等更新内容
- 社交平台：后续可按需接入小红书、微博、V2EX、Reddit 等渠道，补充用户口碑与本地经验

第一版仍建议先做 mock 检索工具和基础 RAG；Agent-Reach 可以作为后续替换真实检索能力的优先方案。

## 5. 用户侧功能需求

## 5.1 旅行需求输入
用户可以通过自然语言输入旅行需求，系统应支持识别：

- 出发地
- 目的地
- 出发日期
- 返回日期
- 出行天数
- 出行人数
- 预算范围
- 交通偏好
- 酒店偏好
- 景点偏好
- 旅行节奏偏好
- 特殊要求与限制条件

支持方式：

- 一次性完整输入
- 多轮逐步补充
- 中途修改条件
- 系统主动追问缺失信息

## 5.2 多轮对话
系统应支持：

- 继承当前会话上下文
- 记住已经确认的旅行条件
- 对局部模块继续追问
- 对已有方案做局部修改或整体重规划

## 5.3 旅行方案生成
系统应生成包含以下模块的方案：

- 交通推荐
- 酒店推荐
- 每日行程安排
- 景点与玩法说明
- 预算估算
- 天气或出行注意事项

## 5.4 方案调整
支持用户进一步提出修改，例如：

- 更改预算
- 更改交通方式
- 更换酒店
- 调整行程节奏
- 增加或删除景点
- 修改某一天的安排
- 重新生成整体方案

---

## 6. 主 Agent 功能需求

Planner Agent 需要具备以下能力：

- 解析用户输入并提取结构化字段
- 判断当前输入属于交通、酒店、行程或综合规划需求
- 根据需求决定调用哪些子 Agent
- 在信息不充分时进行追问
- 汇总多个子 Agent 返回结果
- 发现预算、位置、时间上的冲突并提示
- 统一输出自然语言答复
- 维护 conversation state 与 trip_state

---

## 7. 子 Agent 功能需求

## 7.1 Transport Agent 功能
- 判断用户更适合飞机、高铁或其他交通方式
- 查询交通候选方案
- 支持价格、时间、总耗时、时段偏好等筛选
- 输出多个候选方案和推荐理由
- 支持根据用户反馈重新筛选或替换方案

## 7.2 Hotel Agent 功能
- 识别住宿预算和偏好
- 查询酒店候选数据
- 基于评分、价格、位置、性价比做排序
- 输出多个候选酒店和推荐原因
- 支持对酒店方案做局部替换

## 7.3 Itinerary Agent 功能
- 检索景点与玩法内容
- 根据用户兴趣生成每日行程
- 考虑通勤时间、开放时间、天气与节奏合理性
- 提供景点替换、顺序重排、节奏调整能力
- 让行程具备较好的可执行性与可读性

---

## 8. 状态管理需求

状态系统是整个项目的核心之一。

## 8.1 ConversationState
用于记录对话级上下文，建议包含：

- session_id
- message_history
- summary
- current_intent
- active_agent
- pending_questions
- tool_results
- last_plan
- final_response

## 8.2 TripState
用于记录结构化旅行信息，建议包含：

- origin
- destination
- start_date
- end_date
- duration_days
- budget
- travelers_count
- traveler_type
- transport_preferences
- hotel_preferences
- attraction_preferences
- pace_preference
- must_visit_places
- excluded_places
- notes

## 8.3 状态更新要求
系统应支持：

- 自动从用户输入中提取并更新状态
- 用户修正信息后覆盖旧字段
- 各子 Agent 共享读取 trip_state
- Planner 统一写回状态
- 长对话下通过 summary 压缩上下文

---

## 9. 输出方案需求

最终方案建议采用结构化输出，包括：

- 方案概览
- 交通推荐
- 酒店推荐
- 每日行程
- 预算估算
- 注意事项

可支持不同粒度：

- 简版：快速概览
- 标准版：完整行程与推荐理由
- 详细版：更细的时间安排与候选项说明

---

## 10. 后端与工程需求

## 10.1 API 设计
第一版建议至少提供：

- `/health`：健康检查
- `/chat`：主对话接口
- `/session/{id}`：获取会话状态
- `/plan/{id}`：获取当前方案

## 10.2 服务端能力
- 会话隔离
- 会话恢复
- 日志记录
- 配置管理
- 基础错误处理
- 结果持久化或缓存

## 10.3 推荐技术栈
- FastAPI
- LangGraph
- LangChain
- Pydantic
- Python 异步编程
- 后续可扩展 Redis / MySQL / MCP 服务

---

## 11. RAG 相关需求

RAG 主要服务于 Itinerary Agent，而不是单独形成新 Agent。

主要功能包括：

- 导入旅游攻略内容
- 对攻略进行分块与索引
- 根据目的地和兴趣做检索
- 返回 top-k 内容给行程规划使用
- 对检索结果做基础去重与清洗
- 增强景点推荐与本地玩法建议
- 支持将 Agent-Reach 获取的网页、攻略、公众号、RSS、视频字幕等内容作为 RAG 数据来源

第一版可以做基础 RAG，不必一开始就做复杂 rerank 和多路召回。

建议数据流为：

1. Agent-Reach 负责搜索与读取外部内容
2. 内容清洗模块负责去噪、截断、结构化与来源记录
3. RAG 模块负责分块、索引、召回与去重
4. Itinerary Agent 基于检索结果生成或调整行程

---

## 12. 安全与异常处理需求

系统应具备基础安全与异常兜底能力：

- 输入参数校验
- 工具调用白名单
- 超时与失败兜底
- 当实时数据不可用时，给出说明并降级回答
- 避免虚构高风险信息
- 当预算、时间或路线明显不合理时，主动提示用户

---

## 13. 非功能需求

项目应尽量满足以下工程要求：

- 可扩展：方便增加新工具、新数据源、新 Agent
- 可维护：目录清晰、模块边界明确
- 可测试：支持 mock tools、单元测试、集成测试
- 可运行：最小版本可以本地直接启动
- 可重构：便于后续接入真实 API 和前端系统

---

## 14. 第一版 MVP 范围

为了适合 Claude Code 实战开发，第一版建议聚焦“最小可运行版本”。

### V1 必做
- FastAPI 项目骨架
- `/health` 与 `/chat` 接口
- Planner Agent
- Transport Agent
- Hotel Agent
- Itinerary Agent
- ConversationState 与 TripState
- mock tools
- 基础 RAG 检索
- Agent-Reach 工具适配接口占位
- 方案生成
- 方案调整
- 日志与配置管理
- README 与启动说明

### V1 暂不做
- 前端页面
- 真实第三方平台深度接入
- Agent-Reach 真实渠道的完整配置与自动化调用
- 长期用户偏好记忆
- 复杂推荐排序模型
- 完整权限系统
- 复杂监控与分析后台

---

## 15. 推荐开发顺序

建议按以下顺序推进：

1. 搭建项目骨架
2. 定义 schema 与状态模型
3. 实现 Planner Agent 基础逻辑
4. 实现三个子 Agent 骨架
5. 实现统一 tool interface 与 mock tools
6. 打通 Planner -> 子 Agent -> 汇总输出调用链
7. 增加基础 RAG 检索
8. 增加 Agent-Reach 工具适配接口占位
9. 补充日志、异常处理、README
10. 最后再考虑真实 API / MCP / Agent-Reach 渠道对接

---

## 16. 适配 Claude Code + Superpowers 的开发方式

推荐工作流：

1. 先用 superpowers 做需求梳理与实施计划
2. 再用 subagent-driven-development 分模块执行
3. 每次只推进一个明确模块
4. 先实现 mock 版本，再替换真实能力
5. 先保证能跑通，再逐步重构优化

核心原则：

- 不一次性生成整个项目
- 不过早接入真实第三方服务
- 先定义状态，再定义 Agent，再定义工具，再搭图流程

---

## 17. 总结

该项目的最终目标不是做一个普通聊天机器人，而是做一个具备任务拆解、外部能力调用、多模块协同和结构化状态管理的 AI Agent 应用。

当前架构已经明确为：

- 1 个主 Agent：Planner
- 3 个子 Agent：Transport、Hotel、Itinerary
- 天气、地图、搜索、RAG、Agent-Reach 外部检索等通过 tool / MCP 接入

在工程实施上，应优先完成最小可运行版本，以便后续继续扩展真实数据接入、复杂策略优化与前端展示。
